### 一、完整流程链路（按执行顺序）
#### 1. 入口：用户输入
用户的所有请求（文本、语音、图片/截图）都从这里进入，是整个系统的唯一入口。

#### 2. 并行双分支处理
用户输入会同时走两条独立的处理链路：

##### 🔹 分支1：意图识别 + 内容处理链路（核心问答逻辑）
```
用户输入 → 意图识别 → 分3类意图走不同处理路径 → 汇总到结果回复
```
- **意图识别**：这是系统的"路由器"，自动判断用户请求的类型，比如：
  - 意图一：资讯类/实时类问题（比如"今天天气"、"最新新闻"）→ 走「网络搜索插件」，调用外部搜索获取实时信息
  - 意图二：知识库类问题（比如"产品手册"、"内部规则"）→ 走「RAG（检索增强生成）」，从本地知识库匹配答案
  - 其他意图：闲聊/通用问答（比如"你好"、"1+1等于几"）→ 走「不依赖外部工具的常规回答」，直接用大模型原生能力回答

##### 🔹 分支2：用户画像 + 个性化处理链路
```
用户输入 → 数据库查询 → 提取用户信息 → 影响最终回复策略
```
- 只要用户发请求，就自动查用户数据库，拉取这个用户的画像信息（比如用户的历史对话、偏好、身份标签、历史问题记录）
- 用户信息会同步给最后的「结果回复」模块，用来做个性化调整

#### 3. 结果生成与闭环
```
内容处理结果 + 用户信息 → 根据用户信息决定回复策略 → 结果回复 → 更新用户信息 → 回流到用户数据库
```
- 不是直接输出答案，而是**先结合用户画像做个性化调整**：比如对新用户回复更详细、对老用户可以省略基础说明、对专业用户可以用术语、对普通用户讲大白话
- 回复完成后，自动把本次对话内容更新到用户信息里，用户画像会随着对话不断迭代完善，形成闭环

---

### 二、这个架构的核心设计亮点
1. **解耦设计**：把"内容处理"和"用户个性化"完全拆成两条并行链路，改意图逻辑不影响用户画像，改用户策略不影响问答能力
2. **可扩展性极强**：意图可以无限新增（比如加"代码意图"走代码解释器、加"画图意图"走生图工具），只需要加新的菱形分支就行
3. **越用越准**：用户信息是闭环更新的，用户聊得越多，系统对他的理解越深，回复越贴合需求
4. **兼容多模态**：这个架构天然支持截图/图片输入——用户输入里包含图片时，意图识别可以新增"识图意图"，走视觉模型处理，其他链路完全不用改
